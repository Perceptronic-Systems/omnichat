# Omnichat LLM Interface

Omnichat is an advanced local Large Language Model (LLM) interface designed for sophisticated, interactive chat sessions. The application facilitates Retrieval Augmented Generation (RAG), file system tool access, and complex AI workflows entirely within a private, local environment.

## 🚀 Features

- **Advanced Conversational AI**: Interact with the LLM via a seamless web interface.
- **Local LLM Integration**: Uses Ollama for running various open-source models (e.g., Gemma).
- **Retrieval Augmented Generation (RAG)**: Ability to process and reference local knowledge base documents stored in `/etc/omnichat_knowledge_base`.
- **Multi-Modal Context**: Supports file uploads, including PDF parsing (extracting text), image handling (vision assets), and raw code files.
- **Tool Use Capability**: Implements specialized tools for web search (`search_web`), mathematical computation (`evaluate`), and directory navigation (`list_files`, `read_file`).

## 🛠️ Architecture Overview

The application utilizes a microservices architecture managed by Docker Compose:

1.  **Omnichat Backend (`omnichat-backend`)**: The core API layer built with FastAPI/Uvicorn. It handles the user request, interacts with the LLM client, parses file inputs, and manages tool calling logic before generating responses.
2.  **Ollama Service (`ollama`)**: Provides the local LLM inference endpoint necessary for all AI processing.
3.  **Nginx Proxy (`nginx`)**: Exposes the service via HTTPS (recommended setup).

## 🚀 Setup and Installation

The project requires Docker and Docker Compose to run the integrated services.

### Prerequisites

- Docker Engine installed.
- Docker Compose installed.

### Build and Run Container Services

Running the entire application environment involves the `docker-compose` command, which builds and links all required containers:

```bash
docker-compose up --build -d
```

**Notes:**

- The local knowledge base data must be mounted into the container volume: `volumes: - ./knowledge_base:/etc/omnichat_knowledge_base`.
- This setup requires GPU capabilities for optimal Ollama performance, which is configured in `docker-compose.yaml`.

## 📂 File Structure and Configuration

### Python Dependencies (`requirements.txt`)

The backend relies on a substantial set of libraries for its functionality, including:

- `fastapi`, `uvicorn`: Web API framework components.
- `ollama`: Client library to interact with the Ollama service.
- `pypdf`: Library used for reading and extracting text content from PDF files.
- `sympy`: Used in the `evaluate` tool for mathematical calculations.
- `requests`, `beautifulsoup4`: Libraries required for web scraping within the search tools.

### Backend Logic (`main.py` & `model_gateway.py`)

The core logic resides in these files:

- **`main.py`**: Defines the FastAPI application endpoint `/generate`. This function processes incoming requests, handles file uploads, and streams the response from the LLM to the client using Server-Sent Events (SSE).
  - Handles middleware configuration for CORS access.
  - Implements `StreamingResponse` to manage real-time token generation.

- **`model_gateway.py`**: Encapsulates model interaction logic (`llm` class). It manages the session history, constructs the message payload (including attachments/files), and handles complex LLM calls:
  - Retrieves configuration details from `config.toml`.
  - Parses uploaded files into different content types (PDF text, image bytes, raw text).
  - Implements the tool-use loop logic, catching function calls generated by the model (`tool_calls`) and executing them against defined tools before generating the final response.

### Tool Definitions (`mcp_server.py` & `web_search.py`)

Tool functions are registered using a custom MCP (Master Control Protocol) wrapper for integration with the LLM:

- **`mcp_server.py`**: Contains definitions for available tools, mapping them to executable functions:
  - `list_files(path)`: Lists contents of the knowledge base directory.
  - `search_for_file(filename_query, limit)`: Performs fuzzy search on filenames within the knowledge base.
  - `read_file(filepath)`: Reads and sanitizes content from a specified file path.
  - `search_web(query, limit)`: Calls `search_searxng` to execute external web searches.

- **`web_search.py`**: Contains the dedicated logic for web search functionality:
  - `search_searxng(query, limit)`: Interacts with the SearXNG API to gather title/URL results.
  - `_extract_snippet(...)`: Performs deep scraping on the fetched URLs to extract a contextually rich snippet centered around user query keywords.

### Configuration Files

**`.env` (Recommended)**: Used for environment variables, such as `OLLAMA_HOST`.

**`docker-compose.yaml`**: Defines the complete deployment stack, ensuring necessary network isolation and volume mounting for persistent data storage (`ollama_data`, `/etc/omnichat_knowledge_base`).

```yaml
services:
  omnichat-backend:
    build: .
    # ... (details)
  ollama:
    image: ollama/ollama:latest
    # ... (details for GPU acceleration and service hosting)
```
